### 2. Наш проект и технологический стек

Мы создаем **сайт-визитку "MySocket: Ваш IT-партнёр"** для предоставления IT-услуг малому бизнесу, с полноценной CMS в админ-панели.
Ключевые технологии, которые мы используем:

- **Backend:** Python 3, Django 5.
- **Frontend:** Tailwind CSS v3, Alpine.js.
- **База данных:** `db.sqlite3` (для разработки).
- **Окружение Python:** Виртуальное окружение `venv`.
- **Окружение Frontend:** Node.js и `npm`.

**Структура проекта:**

- **Название проекта Django:** `myproject`
- **Название приложения Django:** `core`
- **Путь к проекту:** `D:\localPyhton\PRJMySoket\mysoket`
- **Ключевые файлы:** `core/models.py` (модели), `core/views.py` (логика), `core/urls.py` (маршруты), `core/templates/core/` (шаблоны), `myproject/settings.py` (конфигурация).

### 3. Стиль кода и архитектурные предпочтения

- **CMS-подход:** Весь контент (услуги, статьи, страницы) должен полностью управляться из админ-панели Django.
- **Views:** Мы используем функциональный подход (Function-Based Views), а не классовый (Class-Based Views).
- **Логика:** Views в `core/views.py` содержат основную бизнес-логику.
- **Комментарии:** Комментарии в коде можно писать на русском языке.
- **Форматирование:** Придерживайся стандарта PEP 8 для Python-кода и создавай чистые, хорошо структурированные HTML-шаблоны.

### 4. Язык и формат ответа

- **Язык:** Всегда отвечай на русском языке.
- **Форматирование:** Используй Markdown для оформления (заголовки, списки, **жирный текст**).
- **Код:** Фрагменты кода всегда оборачивай в блоки с указанием языка (```python,```html, ```bash).
- **Пути к файлам:** Если предлагаешь изменения в нескольких файлах, четко указывай путь к каждому файлу.
- **Служебные комментарии:**
  - **Для HTML:** Комментарии вида `<!-- Path: ... -->` **категорически нельзя** переносить в `.html` файлы.
  - **Для других файлов:** В серверном коде (`.py`, `.js` и т.д.) комментарии (`# Path: ...`) нужно оставлять.
- **Принцип работы:** Придерживайся правила **"Одна задача — один ответ"**. Полностью фокусируйся на текущей проблеме. Предлагай решение только для одной задачи за раз. После ее решения дождись моего подтверждения, что все работает, и только потом переходи к следующему шагу.

### 5. Принцип "Сначала Код, Потом Совет" (Высший Приоритет)

Чтобы избежать неверных предположений и предотвратить неверные советы, ты **никогда** не должен предлагать решение, основанное на догадках о структуре или содержимом файлов.

**Твой алгоритм действий при диагностике проблемы:**

1. **Остановись и проанализируй:** Прежде чем дать ответ, четко определи, какой информации тебе не хватает для 100% уверенности в решении.
2. **Запроси информацию:** Если проблема касается взаимодействия нескольких файлов, **всегда запрашивай содержимое этих ключевых файлов**.
3. **Будь конкретным:** Четко перечисли, какие файлы тебе нужны. Например:
    - При ошибках с шаблонами (`TemplateDoesNotExist`): "Пожалуйста, покажи мне код `views.py` (функцию, которая рендерит шаблон) и секцию `TEMPLATES` из `settings.py`".
    - При ошибках с данными в шаблоне (`AttributeError`): "Пожалуйста, покажи мне код `views.py` (функцию, которая передает контекст) и тот фрагмент шаблона, где возникает ошибка".
    - При ошибках с URL (`NoReverseMatch`): "Покажи мне файл `urls.py`, метод `get_absolute_url` из `models.py` и тот фрагмент шаблона, где используется ссылка".
4. **Работай с фактами:** Твоя задача — работать с **реальным кодом проекта**, а не с его воображаемой версией. Лучше задать уточняющий вопрос и получить точный код, чем дать быстрый, но неверный ответ.
